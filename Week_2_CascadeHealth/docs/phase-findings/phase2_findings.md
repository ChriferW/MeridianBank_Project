# Phase 2 Findings — ClaimsIntake DU Setup

**Date:** 2026-04-29
**Phase:** 2 (Document Understanding setup)
**Status:** Complete
**Author:** Chris Williams

This note captures the work, the findings, and the design decisions for the Document Understanding portion of the ClaimsIntake build. It is the source of record for the DU section of the Phase 7 governance packet and for the Week 3 reflection's coverage of training-data strategy and model evaluation.

## What was built

A trained Modern DU project named `ClaimsIntake_DU` containing one document type, `CascadeClaim`, that extracts 8 scalar fields and one 4-column table from semi-structured Cascade Health claim form PDFs.

The trained model achieved 100% extraction accuracy on every field across the 16 annotated training pages. Validation against an unseen document from the same layout family (CHC-100002) returned predictions that matched the source PDF exactly. This is the gating evidence that Agent 1 in Phase 3 can rely on DU's structured output.

## Test data

Forty PDFs were generated by a parametric script (`generate_claims.py`) modeled on Week 1's SSN-prefix pattern. Claim ID prefix encodes expected adjudication outcome:

- Prefix `1` (14 docs): AUTO_APPROVE family. Routine in-network low-dollar.
- Prefix `2` (10 docs): AUTO_DENY family. Policy exclusions, expired coverage, cosmetic.
- Prefix `3` (7 docs): REVIEW family. Mid-dollar with ambiguous medical necessity.
- Prefix `4` (5 docs): ESCALATE family. High-dollar (>$10K) with REVIEW disposition.
- Prefix `5` (4 docs): LOW_CONFIDENCE family. Deliberately corrupted fields.

Of the 40, 36 (the four families minus LOW_CONFIDENCE) were uploaded to DU. The 4 LOW_CONFIDENCE docs were intentionally held out of training so that DU does not learn to compensate for the corruption patterns. They will be used in Phase 5 to validate confidence-driven routing through Maestro to the RPA fallback.

Full scheme is in `claims_data_scheme.md`.

## Schema

Eight scalar fields plus one 4-column table. Pragmatic schema, not exhaustive. Extracts only what Agent 2 needs to render APPROVE/DENY/REVIEW plus narrative context for the MCP-Claude judgment path.

| Field | Content type | Source location |
| ---- | ---- | ---- |
| claim_id | String | Header (top right) |
| member_id | String | Section 1 |
| date_of_birth | Date | Section 1 |
| policy_number | String | Section 2 |
| provider_npi | ID Number | Section 3 |
| provider_name | String | Section 3 |
| total_charges | Number | Section 5 |
| provider_narrative | String | Section 6 (multi-line) |
| service_lines (table) | — | Section 4 |
| └ date_of_service | Date | column |
| └ cpt_code | String | column |
| └ icd10_code | String | column |
| └ charges | Number | column |

Type rationale: identifier-shaped values with hyphens or letter prefixes were assigned String rather than ID Number because ID Number validates as numeric and rejects values like `CHC-100001` or `CH-PPO-2026-83617`. NPI is exactly 10 digits with no separators, the canonical case for ID Number. CPT and ICD-10 codes are String for the same reason. Currency was not offered in the type vocabulary (the available types are String, Number, Date, Phone, ID Number), so dollar fields use Number. DU's Number type strips the dollar sign during extraction.

Fields deliberately omitted: Section 1 patient demographics other than DOB (last name, first name, sex, address, phone), Section 2 group number and plan type, Section 3 tax ID and provider address, Section 4 line number / POS / description / units, Section 5 amount paid and balance due, Section 7 signature and date. Rationale: Agent 2 doesn't need these for v1 adjudication logic, and adding them increases annotation burden and noise. They can be added in Phase 8 if a real workflow surfaces a need.

## Labeling strategy

Active-learning pattern. Hand-labeled 10 documents covering all four clean scenario families (4 APPROVE, 3 DENY, 2 REVIEW, 1 ESCALATE), then let the trained model predict the rest with manual spot-checks.

The hand-label set was chosen for diversity, not coverage. Different patients, different providers, different table sizes (1 to 3 service lines), different dollar ranges. The premise was that DU only needs to see each scenario family at least twice to generalize the layout, since field positions are identical across all 36 PDFs.

After 10 hand-labels DU surfaced auto-suggestions on subsequent documents. A small number (~6) of high-confidence suggestions were confirmed without correction, expanding the labeled set without meaningful annotation effort. This is documented as "active-learning expansion" in the reflection.

## Auto-training surprise

DU Modern silently triggered training in the background once the labeled set crossed a threshold. No banner, no dialog, no progress indicator at the top of the page — the only signal was the status pill flipping from "Training not started" to "Training in progress" when the user navigated back to the breadcrumb view. Earlier in the session the platform had displayed an explicit minimum-page warning ("Annotate a minimum of 5 pages") that turned out to be the unblocking threshold for training, not the training trigger itself.

This is worth flagging in the governance packet. A platform that silently moves models into a trained state without an explicit operator action has an audit-trail completeness gap: there is no Studio Execution Trail or Orchestrator job entry that documents "training started at HH:MM:SS by user X." The Audit Trail Sample for Phase 7 should note this for the DU model in particular and recommend that production deployments lock training behind an explicit MLOps step (e.g., trigger via Orchestrator job rather than via implicit UI threshold).

## Training results

Project Score: **77 out of 100, rated Good.**

The score is driven by three factors:

| Factor | Result | Rating |
| ---- | ---- | ---- |
| Dataset size | 16 pages annotated, recommended 80–160 | Poor |
| Trained model performance | 100% overall accuracy | Excellent |
| Underperforming fields | Lowest 20% of fields at 100% accuracy | Excellent |

Per-field accuracy at 100% across all 12 fields (8 scalars + 4 table columns).

The Dataset Size "Poor" rating is a calibration warning, not a quality finding. DU calibrates the dataset-size factor against production layouts that vary in form revision, scan quality, and field positioning. Our test corpus does not vary along those axes (all 36 PDFs come from the same generator with identical field positions), so the layout-variance generalization risk that the warning is designed to flag is structurally absent. The reflection writeup will articulate this distinction explicitly: in production, the warning would be a real concern; for this learning challenge, it is not.

## Validation evidence

A single held-out document (CHC-100002, never confirmed by user) was opened post-training. The trained model predicted all 12 fields correctly, matching the source PDF byte-for-byte on every value.

Stronger validation across the remaining 25 unconfirmed clean documents is feasible (open each, eyeball the predictions, capture any errors), but the layout uniformity makes additional spot-checks low-information. One clean prediction on an unseen document is sufficient evidence that the model has learned the layout rather than memorized the training set.

What was deliberately not validated in Phase 2: behavior on degraded inputs. The 4 LOW_CONFIDENCE PDFs (CHC-500001 through CHC-500004, with corrupted DOB, NPI, or CPT fields) were not uploaded to DU. They are reserved for Phase 5 end-to-end testing of the confidence-driven RPA fallback path through Maestro. This is the demo's primary failure-scenario evidence and the reason for keeping those documents out of Phase 2 entirely.

## Decisions locked in

1. **Schema is final for v1.** Eight scalars + 4-column table. No additional fields without a workflow-driven justification.
2. **The 36 uploaded PDFs are the training corpus. The 4 LOW_CONFIDENCE PDFs stay out of DU.** They live in the TestData folder for Phase 5.
3. **CHC-100002 through CHC-400005 (the 25 unconfirmed clean documents) form the validation set.** They will not be confirmed in DU, preserving the audit distinction between training data and validation data.
4. **Dataset Size "Poor" rating is accepted as known calibration noise**, not a remediation target. The reflection will articulate the rationale.
5. **The auto-training behavior is a documented platform finding** and will be flagged in the governance packet's Audit Trail section as a gap requiring compensating control in production.

## Publish and deploy

Project version `v1` was created and toggled to Deployed. Status pill: Deployed. The deployed version is now callable as a Tool by Agent 1 in Phase 3 via DU's Integrate-via-API surface, which exposes a project endpoint URL and the version ID.

After deployment, the version was tagged **Staging** via the Deployment Tag control on the row. The available tag values are Staging and Production. Staging was selected deliberately to demonstrate that environment classification is the right control for distinguishing prototype-grade and production-grade models, and to position the governance packet to articulate "the v1 model is tagged Staging; promotion to Production would require a separate review and retag."

A platform finding worth capturing here. Deployment tag is post-hoc, not required at deploy time. The Deploy toggle and the Deployment Tag are two independent controls. A user can deploy a model with no tag (the row sits with a blank Deployment Tag column until set) and the model is fully callable in that state. The platform also does not block toggling between tags after deployment, so a Staging model can become Production with one click.

Recommended compensating control for the governance packet's MLOps section: deployment of DU model versions should be gated behind an Orchestrator-driven release process where the Orchestrator job is the system of record for which version is "Production" and the underlying DU toggle and tag are treated as primitives, not as release controls. This pairs with the auto-training finding from earlier — together they establish a coherent narrative that UiPath's Modern DU optimizes for time-to-trained-model and time-to-deployed-model at the cost of MLOps discipline that production-regulated workflows would require.

## Reflection seeds

Three threads from Phase 2 worth carrying into the 1500-word reflection.

The active-learning labeling pattern is a real enterprise discipline, not a clever shortcut. Hand-labeling all 36 PDFs would have produced no validation evidence. The 10-then-predict approach produced a measurable baseline accuracy on unseen data, which is the only kind of evidence that can answer the question "does this model generalize?" The reflection should articulate this distinction because it is the kind of judgment that separates a tooling demonstration from an actual ML deployment.

The Dataset Size warning being "Poor" while extraction accuracy was "Excellent" is a perfect case study in calibration mismatch. Default ML quality dashboards are calibrated for production realities (layout variance, scan noise, form revisions) that don't apply to a learning-challenge corpus. The reflection will frame this as the kind of dashboard interpretation skill that distinguishes practitioners who can read a metric in context from practitioners who chase the dial. Both kinds of practitioner exist in real organizations.

The auto-training behavior is the most interesting finding for the governance section. UiPath's Modern DU is at a maturity stage where it optimizes for time-to-trained-model by removing operator friction. That optimization moves a model into a serving state without an audit event. The deployment flow compounds the pattern: deploy is a binary toggle, deployment tag is a separate post-hoc control, and a single click can move a model from Staging to Production with no platform-enforced approval gate. For a learning challenge that's a quirk; for a regulated health insurance workflow the combination of silent training and lightly-gated deployment is a compliance issue. Cascade Health's CRO would not approve a model entering production through a silent UI threshold and a one-click toggle. The Phase 7 governance packet will recommend two compensating controls (training only via explicit Orchestrator job invocation, and deployment promotion gated behind an Orchestrator release process where the Orchestrator job is the system of record for "which version is Production"). The reflection will use this as a concrete example of why agentic platforms still need MLOps discipline layered on top, even when the platform optimizes for "no MLOps required."
